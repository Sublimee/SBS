### Пример 2

Нам предоставили OpenAPI-спецификацию сервиса. Проблема сервиса была в том, что у него было несколько потребителей. Ответы для каждого потребителя должны были бы быть строгими, но для каждого из них в разных полях могли приходить разные по смыслу и свойствам данные. Таким образом для формирования более гибкого API, исходная OpenAPI-спецификация была очень мягкой.

Так мы проговаривали, что определенные поля ответа для нас будут всегда non-nullable (хотя по спецификации они были nullable). Мы завязывались на это в своих DTO и падали c ошибкой, когда приходил некорректный с точки зрения договоренностей ответ.

В коде пришлось делать переход от DTO (OpenAPI-спецификация) к внутренним моделям (договоренности с нами, как с конкретными потребителями). Везде, где между ними были расхождения мы добавляли проверку на nullability (вместо прежнего !!) и выбрасывали понятное исключение или использовали дефолтное значение, если возможно.


### Зависимость от возможности использовать более мягкие реализации, чем гарантирует спецификация

### Пример 1

```kotlin
fun load(): CertificatesDownloadingResult {
    synchronized(loadingLock) {
        if (loadingThread?.isAlive != true) {
            startNewLoading()
        }
        loadingLock.wait()
        return loadingResult ?: CertificatesDownloadingResult.UnknownError
    }
}

private fun startNewLoading() {
    loadingThread = thread {
        val result = interactor.downloadAndVerify()
        synchronized(loadingLock) {
            loadingResult = result
            loadingLock.notifyAll()
        }
    }
}
```

Здесь, если поток в startNewLoading() отработает быстро, т.е. notifyAll() отработает быстрее wait(), то вызывающий поток будет заблокирован. Этого не происходит потому что создание нового потока (thread {...}) обычно отрабатывает долго. Проблема может возникнуть, например, при переходе сервиса на виртуальные потоки.

### Пример 2

В целом всегда будет работать корректно:

```java
@Around("@annotation(LogExecutionTime)")
public Object logExecutionTime(final ProceedingJoinPoint joinPoint) throws Throwable {
    long startTime = System.currentTimeMillis();
    Object proceed = joinPoint.proceed();
    log.debug("{} done in {} ms.", joinPoint.getSignature(), System.currentTimeMillis() - startTime);
    return proceed;
}
```

Однако по спецификации currentTimeMillis может "прыгать", что в System.currentTimeMillis() - startTime может дать даже отрицательное значение. Для таких случаев рекомендуется использовать System.nanoTime().

# Вывод

Спецификация должна стоять во главе угла. От нее должен отталкиваться код. Во всех обсуждениях надо как можно дольше оставаться на уровне спецификации и не уходить на уровень кода (как минимум из-за того, что детали реализации обычно сжирают массу времени, но не являются существенными).

Жаль только, что спецификации в Java и Kotlin приложениях как таковой практически никогда нет :). Хотя бы OpenAPI с идеей contract first много где приживается. Что касается непосредственно логики кода, то хорошо, если есть хотя бы документация в каком-то виде или разработчиком оставлены комментарии. Особенной строгости здесь нет, оттого при виде кода иногда сложно понять, а что вообще задумывал его автор. 

Обычно на основе документации еще расставляются аннотации (наподобие @Positive, @Nullable и др.), которые все же задают некоторый контракт, но этого всё так же мало.

В итоге при разработке упор идет на анализ текущей реализации, которая может быть достаточно изменчива. Только благодаря хорошим тестам мы можем понять, что наши изменения на что-то повлияли и начинаем разбираться в причинах. В моих проектах для меня они давно уже стали заменой (отсутствующим или неточным по разным причинам) спецификации и документации: по ним можно понять, как и когда работает сервис. Но тесты должны быть, действительно, хорошими.
