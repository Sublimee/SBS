# Пример 1

Было:

На вход подавались обычные объекты и объекты-заглушки, в которых первые 4 байта были заранее подготовлены, чтобы их в дальнейшем можно было распознать (могут быть ложные срабатывания). 

```kotlin
// Предусловие:
// На вход подается non-nullable буффер байтов
fun isDumb(buf: ByteBuffer): Boolean {
    val pos = buf.position()
    if (pos !=0 || buf.remaining() <= Wire.MAGIC_SIZE) return false

    // PERF: unrolled magic check (avoid loop overhead)
    if (buf.get(0) != Wire.MAGIC_BYTE_0) return false
    if (buf.get(1) != Wire.MAGIC_BYTE_1) return false
    if (buf.get(2) != Wire.MAGIC_BYTE_2) return false
    return buf.get(3) == Wire.MAGIC_BYTE_3
}
// Постусловие:
// неизмененный исходный буфер
// результат -- ответ на вопрос: содержит ли исходная последовательность байт, начиная с нулевой позиции magic-последовательность, совпадающую с последовательностью байт в объекте-заглушке
```

Стало:

На вход подаются обычные объекты и объекты-заглушки, в которых первый прочтенный байт заранее содержит признак заглушки/обычного объекта, чтобы в дальнейшем их можно было гарантированно отличить.

```kotlin
// Предусловие:
// На вход подается non-nullable буффер байтов, где первый доступный для чтения байт отвечает за признак байт-заглушка / обычный объект
fun isDumb(buf: ByteBuffer): Boolean {
    if (buf.remaining() <= 1) return false

    return buf.get(buf.position()) == Wire.DUMP_BYTE
}
// Постусловие:
// неизмененный исходный буфер
// результат -- ответ на вопрос: является ли объект заглушкой
```

Хоть мы и ослабили требование к buf в коде с точки зрения начальной позиции, это не отразилось на предусловии, так как результат работы метода false такой же равнозначный, как и true и исходный код предусматривал соответствующую проверку. Тем не менее мы дали возможность работать не только с началом буфера. Это может быть полезно, если интересующие нас байты были предворены другими байтами (для каких-то других задач).

В то же время пришлось усилить предусловие, чтобы усилить постусловие. Ложные срабатывания исходной реализации теперь исключены, так как все объекты имеют префикс/суффикс, который мы и проверяем.


# Пример 2

# Было:

Проверкой предусловия нужно было заниматься снаружи каждый раз, когда вызывается метод.

```kotlin
// Предусловие:
// На вход подается non-nullable существующий в системе тип бонусного счета
suspend fun getBonusAccountFaq(
    accountType: String,
): BonusAccountFaqResponse? {
    val loyaltyProgram = LoyaltyProperties.loyaltyProgramTypes[accountType]!!
    return ...
}
// Постусловие:
// результат -- FAQ по переданному типу бонусного счета
```

# Стало:

Ослабляем предусловие, выполняя проверку внутри. Из-за этого ослабляется постусловие:

```kotlin
// Предусловие:
// На вход подается non-nullable тип бонусного счета
suspend fun getBonusAccountFaq(
    accountType: String,
): BonusAccountFaqResponse? {
    val loyaltyProgram = LoyaltyProperties.loyaltyProgramTypes[accountType] ?: throw new AccountTypeNotFoundException(accountType)
    return ...
}
// Постусловие:
// результат -- FAQ по переданному типу бонусного счета или исключение, если бонусного счета не существет
```

Альтернативный вариант, который с одной стороны усиливает постусловие, а с другой -- изменяет его (теперь потребителям 
нужно обрабатывать эту ситуацию явно) -- возвращать пустой faq при неизвестном accountType.

# Пример 4

Поработаем с реализацией отсортированного списка.

# Было:

```java
public class OrderedList<T> {

    ...

    // Предусловие:
    // На вход подаются 2 non-nullable аргумента одного типа: Integer или String
    public int compare(T v1, T v2) {
        if (v1 instanceof Integer && v2 instanceof Integer) {
            return ((Integer) v1).compareTo((Integer) v2);
        }
        if (v1 instanceof String && v2 instanceof String) {
            return ((String) v1).trim().compareTo(((String) v2).trim());
        }
        throw new ClassCastException();
    }
    // Постусловие:
    // Результат: целое число
    // < 0,   если v1 <  v2
    // = 0,   если v1 == v2
    // > 0,   если v1 >  v2
    ...
```

# Стало:

```java
public class OrderedList<T> {

    private final Comparator<T> comparator;
    ...

    // Предусловие:
    // На вход подаются 2 non-nullable аргумента одного типа
    public int compare(T v1, T v2) {
        int result = comparator.compare(v1, v2);
        return ascending ? result : -result;
    }
    // Постусловие:
    // Результат: целое число, результат сравнения
    // < 0,   если v1 <  v2
    // = 0,   если v1 == v2
    // > 0,   если v1 >  v2
    ...
```

Обратная совестимость не нарушена, так как ослабили предусловие, а постусловие осталось такое же (все за счет изменения 
реализации). Такие самописные библиотечные классы -- обычно первый кандидат для того, чтобы расширить их область 
применения.


# Вывод

Самый главный инсайт: ослабление предусловия и при этом усиление постусловия -- тот случай, когда можно побороть 
Open-Closed Principle, изменив исходный код!

Это высший пилотаж, который мне в полной мере не поддался: ни в одном моем примере одновременно это сделать не 
получилось.

Ослабить предусловие часто несложно: просто добавляем соответствующие проверки внутрь метода (1) или расширяем
область допустимых входных аргументов (2). По (1) стоит скзать, что ослабление предусловий приводит к тому, что методом 
становится неудобно пользоваться: нужно было каждый раз проверять соблюдение предусловий в вызывающем коде, что скорее 
неудобно для пользователей, который не хотят разбираться в нюансах спецификации. Это требует высокой осознанности 
разработчиков.

Попытка ужесточения постусловий приводила скорее к их изменению и нарушению обратной совместимости. В то же время 
ослабление предусловий приводила к ослаблению постусловий.

Кажется, что максимальный эффект можно достичь на библиотечном коде, когда мы хотим предоставить максимальную 
универсальность, но при этом точный результат. Пример 1 это хорошо показал, там идея обобщения обработки разных объектов
стоит во главе угла.
